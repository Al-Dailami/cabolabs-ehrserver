binding request parameters to action method parameters

73 posts
One of the new features in Grails 2 will be a mechanism to automatically bind request parameters to controller action method arguments.  The idea is to support something like this:

class DemoController {

    def someAction(String name, int age) {
        // ...
    }

}

In Grails 1.3 that might be something like this:

class DemoController {

    def someAction = {
        String name = params.name
        int age = params.int('age')
        // ...
    }
}

Some of the work has begun to start adding support for that.  MethodActionTransformerParameterBindingSpec describes the current behavior.  Right now there is support for String and 6 of the 8 primitive types (no byte and no char) and their corresponding type wrappers.  The impl is still funky but I am mostly interested in defining desired behavior at this point.  If you care to look at MethodActionTransformerParameterBindingSpec, please share any thoughts or concerns.

Thanks for your time.

Re: binding request parameters to action method parameters

70 posts
This looks good for the simple case.  It feels very similar to Spring's @RequestParam.  One thing that we lose is the ability to map a param 'age' to a differently named someAction(int studentAge) in Spring you could annotate the method parameter like this:
@RequestParam("age") int studentAge
In Grails 1.3.7, you would just write this in the closure:
int studentAge = params.int('age')
Would be handy to have an additional (optional) mechanism to map params to method parameters.

What about non primitive types?  Would they be databound the same way?

def someAction(Book book) { ... }

Where does this leave Command Objects?

def someAction(MyCommand commandBean, String name, int age)  { ... }


There are a few other areas of consideration:
  • How do Custom Property Editors fit into this?
  • How does validation work on these (as Graeme pointed out)

On Thu, Jun 30, 2011 at 11:59 AM, Colin Harrington
<[hidden email]> wrote:

> This looks good for the simple case.  It feels very similar to Spring's
> @RequestParam.  One thing that we lose is the ability to map a param 'age'
> to a differently named someAction(int studentAge) in Spring you could
> annotate the method parameter like this:
>
> @RequestParam("age") int studentAge
>
> In Grails 1.3.7, you would just write this in the closure:
> int studentAge = params.int('age')
> Would be handy to have an additional (optional) mechanism to map params to
> method parameters.
... [show rest of quote]

At the moment the mapping from request parameter name to method
argument name is by convention.  There is no configuration option.  If
for some reason I want a local variable with a different name, I can
declare and initialize it as you show above.  That of course will
still work in Grails 2.  The convention based approach will surely
satisfy many use cases.  If there is enough interest in arbitrarily
mapping request parameter names to method argument names we would work
something like that up.  I question the usefulness.  The 80/20 rule
may be more like 90/10 in this case.  What we have is a great option,
even if it didn't evolve much further (which it will).


>
> What about non primitive types?  Would they be databound the same way?
>

Right now there are 9 non-primitive types supported.  Those are the 8
java.lang type wrappers and java.lang.String.  We should probably have
support for some other basic types (Date comes to mind) and maybe a
mechanism for plugging in converters for arbitrary types.  Some of
that might be useful and I wouldn't say that is necessarily out of
scope, but is beyond what I wanted to knock out with this first step
yesterday.

>
> Where does this leave Command Objects?
>
> def someAction(MyCommand commandBean, String name, int age)  { ... }

Command object binding and validation should continue to work as it
has.  Depending on what we end up with for validation options, some of
this may or may not overlap with what you might have otherwise used
command objects for.

>
>
> There are a few other areas of consideration:
>
> How do Custom Property Editors fit into this?

Custom Property Editors don't play any role at all in the current
implementation.


> How does validation work on these (as Graeme pointed out)
>

There is no validation at all in the current implementation aside from
handling conversion failures.  As far as validation goes, the
following are the same:

def grails137Action = {
    int age = params.int('age')
    // ...
}

def grails2Action(int age) {
    // ...
}

There are a number of nice things that become options with this.  For
example, default parameters will work:

def grails2Action(String firstName, int age = 99, boolean
requirePasswordChange = true) {
}

If the "age" and "requirePasswordChange" request parameters are there
and can be converted, their values will be passed in otherwise the
default values are in place.  That isn't something that will apply to
every controller action of course, but certainly some will be made
more simple by that option.

I think this is a great feature and like everything else in the
framework, we want feedback and suggestions from everyone interested.

Thanks!
